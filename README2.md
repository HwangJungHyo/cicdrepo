# cicdrepo

> 본 문서는 이 프로젝트의 **브랜치 전략**, **환경별 배포 전략**, **골든 이미지 파이프라인**에 대한
> **단일 규범(single source of truth)** 입니다. 프로젝트에 기여하는 모든 구성원은 아래 규칙을 따릅니다.
>
> 이 규범의 소유권과 거버넌스는 **DevOps 엔지니어**에게 있습니다. 브랜치 전략과 배포 전략의 결정 권한은
> DevOps 엔지니어가 가지며, 협업 조직의 요구사항을 고려하되 최종 설계·의사결정 책임은 DevOps 엔지니어가 집니다.
>
> **이 문서의 모든 설계 결정에는 근거가 있습니다.** 보안 게이트는 실패 케이스로 검증했으며,
> 검증 과정에서 발견한 구멍과 그 대응을 함께 기록합니다. "설정했다"가 "작동한다"를 보장하지 않기 때문입니다.

## 1. 전체 브랜치 전략 및 CI/CD 프로세스

### 1-1. 브랜치 구성

| 구분 | 브랜치 | 라이프사이클 | 역할 |
|------|--------|--------------|------|
| Long-lived | `main` | 영구 | 통합 브랜치. 항상 배포 가능한 상태 유지 |
| Short-lived | `feature/*`, `fix/*`, `ci/*`, `chore/*` | 하루~이틀 | 개인 작업 브랜치. 머지 후 즉시 삭제 |
| Tag (브랜치 아님) | `qa-*`, `v*` | - | 환경 승격 시점을 고정하는 태그 |

**설계 결정: Trunk-Based Development**

`develop` / `release` 같은 롱텀 브랜치를 두지 않습니다. 환경 승격은 **브랜치가 아니라 태그**로 관리합니다.

**근거:**
- 롱텀 브랜치가 오래 살수록 `main`과의 diff가 벌어져 **머지 충돌(merge hell)** 이 기하급수로 커집니다.
- `develop` 브랜치는 여러 feature를 모았다가 한꺼번에 흘려보내는 **저수지 구조**로, 두 가지 병목을 만듭니다:
  1. **축적 지연** — 개별 기능의 배포가 릴리즈 주기에 인질로 잡힘
  2. **장애 원인 특정 곤란** — 여러 변경이 한꺼번에 나가 어느 것이 원인인지 좁히기 어려움
- GitFlow 창시자(Vincent Driessen)도 2020년 원글에 "continuous delivery를 하는 팀은 더 단순한 워크플로를 쓰라"고 덧붙였습니다.

**언제 이 결정이 틀리는가:** 명시적 버전 관리·다중 버전 동시 지원이 필요한 제품(DB, SDK, 온프렘 설치형, 모바일 구버전 지원)에는 GitFlow가 여전히 정당합니다. 본 프로젝트는 단일 버전 서비스이므로 trunk-based를 채택합니다.

**미완성 기능 처리:** trunk-based는 미완성 코드도 `main`에 들어갈 수 있으므로, **feature flag**로 감싸 배포는 하되 사용자에게 노출하지 않습니다. 이로써 **배포(deploy)와 릴리즈(release)를 분리**합니다.

### 1-2. 전체 CI/CD 흐름

```
[로컬 작업 + 단위 테스트]  — feature 브랜치
        │  git push (feature 브랜치로. main 아님)
        ▼
[Pull Request]
        │  CI(lint·test·CodeQL) + 코드 리뷰 승인
        │  golden base 검증 + 이미지 취약점 스캔
        │  ↑ 두 관문을 모두 통과해야 머지 가능 (branch protection)
        ▼
[main 머지]  ← CD 파이프라인의 첫 트리거 (이벤트: merge)
        ▼
[DEV 자동 배포]  — 통합 테스트(integration test)
        │  담당자가 QA 시작을 판단하고 태그 생성 ← 코드 프리징 지점
        ▼
[QA 배포]  — 재빌드 없이 동일 digest 승격
        │  QA 통과 후 승격
        ▼
[STAGE 배포]  — 운영 유사 환경 리허설
        │  수동 승인(manual approval)
        ▼
[PROD 배포]
```

### 1-3. 관통 원칙

| 원칙 | 의미 |
|------|------|
| **CI ≠ CD** | 머지는 코드가 `main`에 통합된 것일 뿐, 배포가 아닙니다. 머지가 CI의 끝이고 CD의 시작입니다. |
| **Promote, not rebuild** | 환경 승격 시 재빌드하지 않고 **동일 아티팩트(image digest)** 를 이동합니다. |
| **Build once, deploy many** | 이미지는 모든 환경에서 동일하게 유지하고, 환경별 차이는 ConfigMap으로 분리합니다. |
| **불변 인프라** | 실행 중인 파드를 직접 수정하지 않습니다. 변경은 재빌드 → 새 digest → 재배포로만. |
| **사람의 실수를 자동화로 대체** | 환경 간 구조 동기화, 취약점 검사, 배포 트리거를 기계가 수행합니다. |

## 2. 소스 코드 컨트리뷰션 방법

### 2-1. 작업 절차

1. `main`을 최신화합니다. (`git checkout main && git pull origin main`)
2. `main`에서 **feature 브랜치를 새로 땁니다.**
3. 로컬에서 작업하고 **단위 테스트를 통과**시킵니다. (로컬 검증 없이 push하면 다른 사람의 코드를 해칠 수 있습니다)
4. push 후 **Pull Request**를 생성합니다.
5. **CI 통과 + 리뷰어 승인**을 모두 만족하면 `main`에 머지합니다.
6. 머지 후 **feature 브랜치를 삭제**합니다.

> **push ≠ merge.** `git push`는 원격의 feature 브랜치에 올리는 것이고, PR은 "main에 합쳐 달라"는 **요청**입니다.
> 실제로 `main`에 들어가는 것은 **merge** 시점입니다.

### 2-2. 브랜치 네이밍

| Prefix | 용도 | 예시 |
|--------|------|------|
| `feature/` | 신규 기능 | `feature/add-cicd-app` |
| `fix/` | 버그 수정 | `fix/upgrade-go-builder` |
| `hotfix/` | 운영 긴급 패치 | `hotfix/payment-timeout` |
| `ci/` | 파이프라인 변경 | `ci/harden-golden-pipeline` |
| `chore/` | 빌드·설정·문서 | `chore/update-codeowners` |

형식: `prefix/짧은-설명` (kebab-case)

### 2-3. 머지 규칙 (branch protection)

- `main`은 **직접 push 금지**. PR을 통해서만 변경 가능합니다.
- 머지 조건: **CI green + 최소 1인 이상 리뷰 승인**
- 머지 방식: **Squash and Merge** (커밋 히스토리 단순화)

### 2-4. Hotfix 처리

프로덕션 긴급 수정도 **반드시 Git을 통해서만** 이루어집니다.

**금지: 실행 중인 파드/리소스 직접 수정**
- 파드 재시작·오토스케일링·후속 배포 시 수정이 **유실**됩니다.
- 이미지 digest가 변경되지 않아 **배포 검증 체계가 무력화**됩니다.
- 변경 이력 추적이 불가능해집니다(configuration drift).

**우선 대응 (동일 속도, 안전):**
1. 배포 롤백 (`kubectl rollout undo`)
2. feature flag 비활성화 (kill switch)
3. ConfigMap 수정 후 재배포

**긴급 시에도 경로는 같습니다.** 절차를 생략하는 것이 아니라 **처리 속도를 높입니다**(리뷰어 즉시 호출, 필수 검사만 수행).

본 프로젝트는 `main` 단일 브랜치 구조이므로 **hotfix 역병합(back-merge) 문제가 발생하지 않습니다.**
릴리즈 브랜치를 분리하는 구조에서는 hotfix가 `main`에 반영되지 않으면 다음 배포 시 수정이 유실되므로,
cherry-pick 또는 back-merge를 절차적으로 강제해야 합니다.

## 3. 환경별 배포 전략

### 3-1. 요약

| 환경 | 목적 | 트리거 | 배포 대상 | 자동화 |
|------|------|--------|-----------|--------|
| **DEV** | 개발자 통합 테스트 | `main` 머지 이벤트 | `main` 최신 커밋 | 완전 자동 |
| **QA** | QA 조직 기능 검증 | 릴리즈 태그(`qa-*`) 생성 | 해당 태그의 digest | 반자동 |
| **STAGE** | 운영 유사 최종 리허설 | QA 통과 후 승격 | 검증된 digest | 반자동 |
| **PROD** | 실제 운영 | `v*` 태그 + 수동 승인 | 검증된 digest | 승인 후 자동 |

### 3-2. DEV — 머지 이벤트로 완전 자동

**트리거:** `main` 브랜치에 feature 브랜치가 머지되는 이벤트

**왜 자동인가:** 트리거 설계의 판단 기준은 **"이 환경에 배포하는 시점을 사람이 정해야 할 이유가 있는가"** 입니다.
DEV는 개발자 검증용 환경으로 실사용자 영향이 없으며, 배포를 통제할 이유가 없습니다.

**왜 즉시여야 하는가:** 각 개발자의 코드는 개별적으로는 정상이어도 **합쳐졌을 때 깨질 수 있습니다**(semantic conflict).
PR의 CI는 "내 코드 + 그 시점의 main"만 검증하므로, 그 뒤 다른 PR이 머지되면 **실제 합쳐진 조합은 아무도 검증하지 않은 상태**가 됩니다.
머지 단위로 즉시 배포하면 문제 발생 시 **직전 변경이 원인**임이 자명해져 장애 원인 추적 비용이 최소화됩니다.

**특성:** 스프린트 동안 유효 브랜치가 N개 머지되면 DEV에는 N번 배포가 일어납니다. 이는 정상 동작입니다.

### 3-3. QA — 태그로 코드 프리징

**트리거:** 담당자가 "QA를 시작한다"고 판단하고 **릴리즈 태그(`qa-2026.07-rc1`)를 생성하는 순간**

**왜 QA를 DEV에 붙이면 안 되는가 (치명적):**
DEV의 배포 트리거는 머지입니다. QA가 검증하는 도중 다른 개발자가 머지하면 **테스트 대상 코드가 밑에서 바뀝니다.**
통과하던 테스트가 갑자기 깨지고, 버그 재현이 불가능해집니다(moving target).
따라서 QA는 DEV와 **다른 라이프사이클을 가지는 안정 환경**이어야 합니다.

**코드 프리징 구현:** 움직이는 브랜치가 아니라 **고정된 태그**를 배포 대상으로 삼습니다.
- 브랜치는 merge될 때마다 가리키는 커밋이 바뀌는 **살아있는 포인터**
- 태그는 특정 커밋에 박아둔 **고정된 이름표** (뒤에 새 커밋이 쌓여도 안 움직임)

**왜 반자동인가:** "모든 기능이 완료됐다"를 기계가 자동 감지하는 것은 불가능합니다.
따라서 **태그를 생성하는 한 번의 수동 행위**가 프리징 시점을 확정하고, 이후 배포는 자동화됩니다.

| 행위 | 주체 |
|------|------|
| "QA 시작 시점" 판단 → 태그 생성 | **사람** |
| digest 조회 · YAML 치환 · apply · 파드 생성 | 기계 (전부 자동) |

**100% 자동도, 100% 수작업도 아닙니다.** 결정만 사람, 실행은 자동입니다.

**전제 조건:** QA 태그는 **이미 DEV에 배포되어 통합 테스트를 거친 커밋**에만 부여합니다.
빌드된 이미지가 레지스트리에 존재하지 않는 커밋에 태그를 찍으면 파이프라인이 `ImageNotFound`로 실패합니다.
이는 의도된 동작입니다 — 검증되지 않은 코드가 QA로 승격되는 것을 막습니다.

### 3-4. DEV 통합 테스트와 QA 검증은 왜 둘 다 필요한가

| | DEV 통합 테스트 | QA 검증 |
|---|-----------------|---------|
| 주체 | 기계 (자동) | 사람 |
| 관점 | **코드 관점** | **사용자 관점** |
| 잡는 것 | 논리 오류, API 충돌, 회귀 | UX, 기획 의도 불일치, 예상 못 한 조합 |
| 한계 | **작성된 테스트만** 검증 | 느림, 비용 |

**서로 못 잡는 영역이 다릅니다.** 자동 검증으로 빠르게 1차 필터링하고, 사람 검증으로 사용자 경험을 2차 확인하는 이중 구조입니다.

### 3-5. 승격 방식 — Promote, not Rebuild

**원칙:** 각 환경에서 이미지를 다시 빌드하지 않습니다. DEV에서 검증한 **동일 digest**를 그대로 승격합니다.

**왜:** 환경마다 재빌드하면 빌드 시점의 의존성 해결 결과가 달라질 수 있어, **"테스트한 것"과 "배포한 것"이 다른 물건**이 됩니다.

**승격 단위는 태그가 아니라 digest입니다:**

| | 태그 (`:v1.0`) | digest (`@sha256:...`) |
|---|---------------|------------------------|
| 성질 | 사람이 붙인 이름표 | 내용의 해시(지문) |
| 내용이 바뀌면 | 이름은 그대로, 가리키는 것만 바뀜 | **값 자체가 바뀜** |
| 신뢰성 | 같은 태그가 다른 내용일 수 있음 | 같으면 반드시 같은 내용 |

**검증 방법:**
```bash
kubectl get pod -n dev -o jsonpath='{.items[0].status.containerStatuses[0].imageID}'
kubectl get pod -n qa  -o jsonpath='{.items[0].status.containerStatuses[0].imageID}'
# 두 digest가 동일해야 승격이 올바르게 수행된 것
```

### 3-6. 환경 동기화와 설정 분리

| 대상 | 관리 방법 |
|------|-----------|
| 환경 **구조**(동일해야 하는 부분) | **IaC**로 코드화하여 자동 동기화 |
| 환경별 **차이값**(로그 레벨, 리소스, feature flag) | **ConfigMap**으로 분리. 이미지는 동일, 설정만 교체 |
| 비밀값 | **Secret** + 외부 관리(Vault, External Secrets, Sealed Secrets) |

**원칙:** "PROD를 바꾸면 STAGE/QA도 똑같이 바꾸면 된다"가 아니라,
**"사람이 실수할 수 있는 부분은 자동화한다"** 가 정답입니다.

> **ConfigMap 주입 함정:** 환경변수(`envFrom`) 방식은 파드 생성 시 한 번만 주입됩니다.
> ConfigMap을 나중에 만들거나 변경하면 `kubectl rollout restart`가 필요합니다.
> **ConfigMap 먼저 → Deployment 나중** 순서를 지키면 restart가 불필요합니다.
> Kustomize `configMapGenerator`는 내용 해시를 이름에 붙여 자동 rollout을 유발하므로, 이 문제를 구조적으로 해결합니다.

### 3-7. Secret은 암호화가 아닙니다

Kubernetes Secret은 **base64 인코딩일 뿐 암호화가 아닙니다.**
실제 보호는 etcd encryption at rest, Vault, External Secrets Operator, Sealed Secrets 등으로 구현해야 합니다.
ConfigMap에 비밀값을 넣지 않습니다.

## 4. 골든 이미지 파이프라인 관리

### 4-1. 정의와 목적

골든 이미지는 조직이 **보안 검증을 거쳐 승인한 표준 베이스 이미지**입니다.

**골든 이미지 파이프라인 관리는 "버전 관리"가 아닙니다.** 버전 관리만으로는 "무엇을 쓰는지"는 알 수 있으나
"안전한지"는 알 수 없습니다. 이 파이프라인이 관리하는 것은 다음 네 가지입니다:

| 관리 대상 | 답하는 질문 | 장치 |
|-----------|-------------|------|
| **승인** | 어떤 베이스를 써도 되나? | 화이트리스트 + 앱 CI 강제 |
| **안전성** | 취약점이 있나? | Trivy 스캔 게이트 |
| **진위** | 우리가 만든 게 맞나? | Cosign 서명 (미적용, §5 참고) |
| **투명성** | 안에 뭐가 들었나? | SBOM |
| (식별) | 어느 것인가? | semver + 불변 digest |

**정기 재스캔이 이를 증명합니다.** Dockerfile이 하나도 안 바뀌었는데도 매주 파이프라인을 돌립니다.
**같은 버전인데 어제는 안전했고 오늘은 위험할 수 있기 때문**입니다. 버전 관리가 목적이라면 할 이유가 없는 일입니다.

### 4-2. 왜 앱 CI와 분리하는가

| | 앱 코드 | 베이스 이미지 |
|---|---------|--------------|
| 변경 주기 | 하루에도 여러 번 | 몇 주~몇 달 (또는 CVE 발생 시) |
| 트리거 | 코드 merge | 베이스 변경 / **정기 스케줄** |
| 검증 | 단위 테스트 | **취약점 스캔** |

라이프사이클이 다릅니다. 섞으면 어느 앱이 어느 베이스를 쓰는지 추적할 수 없고,
베이스 패치를 앱마다 따로 해야 합니다.

### 4-3. 파이프라인 구성 (다층 방어)

```
1. 브랜치 검증      → main 외에서는 push 금지
2. 화이트리스트     → distroless + digest pin만 허용    ★ 가장 확실 ★
3. 빌드 (로컬만)    → 검증 통과 전에는 절대 push 안 함
4. EOL 차단         → 스캔 데이터가 없는 OS 거부
5. 스캔 게이트      → exit-code 1로 HIGH/CRITICAL 차단
6. SBOM 생성        → CycloneDX
7. ECR push         → 모든 방어를 통과한 것만
+ 주간 정기 재스캔  → 빌드 후 공개되는 신규 CVE 탐지
```

**방어 순서가 핵심입니다.** 스캔은 **마지막 방어선**이지 첫 번째가 아닙니다.
스캐너가 보지 못하는 것이 있으므로(§4-5), 그 앞에서 걸러야 합니다.

### 4-4. 베이스 이미지 정의 원칙

| 설정 | 막으려는 것 | 원칙 |
|------|------------|------|
| `FROM ...@sha256:` (digest pin) | 재현 불가능한 빌드 | 재현성 |
| distroless | 침입 후 공격 확대 | 최소 공격 표면 |
| `USER 65532` (nonroot) | 컨테이너 탈출 시 피해 확대 | 최소 권한 |
| OCI 표준 라벨 | 출처 불명 이미지 | 추적성 |

> distroless 이미지는 셸·패키지 매니저·유틸리티가 없습니다. `kubectl exec ... wget`이 실패하는 것은
> 버그가 아니라 이 설계의 증거입니다. 침입자에게도 똑같이 없습니다.

### 4-5. 검증: 실패 케이스로 확인한 것

**보안 통제는 실패 케이스로만 검증됩니다.** 통과하는 것은 아무 검사도 하지 않아도 통과하기 때문입니다.
아래는 실제로 수행한 실험과 그 결과입니다.

| # | 입력 | 하드닝 전 | 하드닝 후 | 발견 |
|---|------|-----------|-----------|------|
| 1 | distroless (digest pin) | 통과 | 통과 | 정상 동작 |
| 2 | **ubuntu:20.04 (EOL)** | **통과 (뚫림)** | 차단 (화이트리스트) | **스캐너 맹점** |
| 3 | node:16 (취약) | 차단 (스캔) | 차단 (화이트리스트) | 게이트 정상 |
| 4 | 앱 CI: `FROM ubuntu:20.04` | (검사 없음) | 차단 (베이스 강제) | 강제 장치 작동 |

#### 발견 ①: EOL 이미지는 스캔을 0건으로 통과합니다

```
$ trivy image --severity HIGH,CRITICAL ubuntu:20.04
WARN  This OS version is no longer supported by the distribution
WARN  The vulnerability detection may be insufficient
Vulnerabilities: 0
```

**원인:** EOL 배포판은 보안 공지를 발행하지 않으므로 스캐너 DB에 CVE 데이터가 없습니다.
**`0건`은 "안전"이 아니라 "탐지 불가"입니다.**

이것은 스캔 옵션 조정으로 해결되지 않습니다(`--ignore-unfixed` 유무와 무관하게 0건).
Trivy는 경고를 출력했으나 파이프라인이 이를 무시했습니다.

**대응:**
1. **베이스 화이트리스트** — 스캔 이전에 차단. 스캐너의 지식에 의존하지 않으므로 EOL이든 zero-day든 무관하게 막습니다.
2. **EOL 경고를 실패로 승격** — Trivy의 `no longer supported` 경고를 exit 1로 전환

> **주의:** 같은 EOL이라도 debian 10.13(node:16의 기반)은 239개 취약점이 탐지되었습니다.
> "EOL이면 0건"이라는 규칙조차 성립하지 않으므로, **EOL 경고 자체를 차단 신호로 삼아야** 합니다.

#### 발견 ②: `workflow_dispatch` 기본값이 취약 이미지에 정식 태그를 부여했습니다

`version` 입력의 `default: '1.0.0'` 때문에, 실험 브랜치에서 무심코 실행한 결과
**ubuntu:20.04 이미지가 `1.0.0`이라는 가장 신뢰받는 태그로 골든 리포지토리에 등록**되었습니다.

**대응:** `default` 제거 + **main 브랜치 외 push 금지** + 비-main은 `test-` 접두사 강제 (삼중 방어)

> **태그 불변성(IMMUTABLE)의 한계:** 리포지토리를 IMMUTABLE로 설정했음에도 이 사고가 발생했습니다.
> IMMUTABLE은 "같은 태그의 재지정"을 막지만 "새 태그로 취약한 이미지를 push하는 것"은 막지 않습니다.
> **태그 불변성은 무결성 장치이지 품질 장치가 아닙니다.**

#### 발견 ③: 골든 베이스가 깨끗해도 앱은 취약할 수 있습니다

앱 CI 스캔 게이트가 발견한 결과:

```
app:candidate (debian 13.5)   debian     0     ← 골든 베이스: 깨끗
app                           gobinary   15    ← Go 바이너리: CRITICAL 1, HIGH 14
```

**원인:** 빌드 스테이지의 `golang:1.21`이 컴파일한 바이너리에 취약한 stdlib(v1.21.13)이 정적 링크되었습니다.
`CVE-2025-68121` (crypto/tls: 세션 재개 시 인증서 검증 오류, CRITICAL)

**핵심 교훈:**
- **골든 이미지는 OS 레이어만 보장합니다.** 앱이 무엇을 링크하는지는 별개입니다.
- **digest pin은 재현성을 주지만 안전성을 주지 않습니다.** `golang:1.21@sha256:...`는 digest로 고정되어 있었으나, 그 버전이 낡았다는 것은 아무도 확인하지 않았습니다.

**대응:** 빌드 이미지를 `golang:1.25`로 갱신 (15건 → 0건). 근본 대응은 §5 참고.

#### 발견 ④: `if: always()`가 진짜 원인을 가립니다

`Trivy scan (SARIF)` 단계에 `if: always()`를 걸어둔 결과, 빌드가 실패한 경우에도 스캔이 실행되어
`unable to find the specified image` 라는 **후속 에러가 로그를 덮었습니다.**

**교훈:** 에러 설계도 파이프라인 설계의 일부입니다. 연쇄 에러는 원인 추적을 방해합니다.

### 4-6. 앱 CI의 강제 장치

골든 이미지를 만들어두는 것만으로는 **아무도 쓰라고 강제하지 않습니다.**
개발자가 다른 베이스를 사용해도 막을 방법이 없다면, 골든 이미지는 "권장 사항"이지 "규칙"이 아닙니다.

앱 CI(`build-push.yml`)는 다음 두 게이트를 수행합니다:

| 게이트 | 검사 대상 | 막는 것 |
|--------|-----------|---------|
| **베이스 검사** | Dockerfile의 런타임 `FROM` | 승인되지 않은 베이스 사용 (digest pin 강제) |
| **앱 이미지 스캔** | 빌드된 최종 이미지 | 앱이 추가한 취약점 |

**둘 다 필요합니다.** 베이스 검사만으로는 앱이 추가한 취약점(발견 ③)을 못 잡고,
스캔만으로는 EOL 베이스(발견 ①)를 못 잡습니다.

### 4-7. 취약점 예외 정책 (`.trivyignore`)

이 파일의 목적은 **취약점을 숨기는 것이 아니라, 인지하고 의식적으로 수용했음을 근거와 함께 기록하는 것**입니다.

**규칙:**
1. 모든 예외에 **만료일(`exp`) 필수.** 만료일 없는 예외는 "영원히 수용한다"는 선언과 같습니다.
2. 사유·근거·승인자를 주석으로 남깁니다.
3. 만료일이 지나면 Trivy가 다시 탐지 → 파이프라인 실패 → **재검토가 강제**됩니다.

```
CVE-2026-12345 exp:2026-09-30
  사유: 우리 런타임에서 사용하지 않는 코드 경로에 존재
  근거: 앱이 해당 라이브러리의 XML 파서를 호출하지 않음 (코드 리뷰 확인)
  승인: 보안팀 (2026-07-09)
```

**왜 예외 장치가 필요한가:** 예외가 없으면 개발자들은 파이프라인에 막힐 때마다
**게이트 자체를 꺼버립니다.** 그러면 모든 CVE가 통과합니다.
예외 장치는 **게이트를 살리기 위해** 존재합니다. "완벽한 통제"보다 "지속 가능한 통제"가 실무의 답입니다.

같은 이유로 `--ignore-unfixed`를 사용합니다. 패치가 존재하지 않는 CVE까지 차단하면
빌드가 영원히 통과하지 못하고, 팀은 결국 게이트를 우회합니다.

## 5. 알려진 한계와 개선 계획

**이 파이프라인은 완성되지 않았습니다.** 아래는 인지하고 있으나 아직 구현하지 않은 항목입니다.
알려진 약점을 기록하는 것도 설계의 일부입니다.

### 5-1. 미구현 (우선순위 순)

| 우선순위 | 항목 | 없으면 발생하는 위험 |
|----------|------|---------------------|
| **높음** | **Admission Controller** (Kyverno/OPA) | CI를 우회한 배포(수동 `kubectl apply`)를 막을 수 없음. 최후 방어선 부재 |
| **높음** | **빌드 이미지 거버넌스** | 런타임 베이스만 통제 중. 빌드 스테이지는 수동 관리 (발견 ③) |
| 중간 | **Cosign 서명 및 검증** | 이미지의 진위(우리가 만든 것인지)를 확인할 수 없음 |
| 중간 | **IAM 역할 분리** | 앱 CI 역할이 골든 이미지 push 권한도 보유. 앱 CI 탈취 = 골든 이미지 오염 |
| 중간 | **앱 이미지 정기 재빌드** | Go stdlib CVE는 지속적으로 공개됨. 앱 Dockerfile이 안 바뀌어도 취약해짐 |
| 낮음 | CODEOWNERS + 승인 게이트 | 골든 이미지 변경에 보안팀 승인 절차 부재 |
| 낮음 | SBOM 장기 저장·조회 | artifact 90일 보존. 과거 이미지의 구성요소 조회 불가 |
| 낮음 | ECR lifecycle policy | 이미지 무한 축적 |
| 낮음 | 알림 연동 (Slack/PagerDuty) | 정기 스캔 실패를 즉시 인지 불가 |

### 5-2. 가장 큰 구멍: 강제 지점이 CI에만 있습니다

현재 골든 이미지 사용 강제는 **CI 파이프라인에서만** 이루어집니다.
CI를 거치지 않고 클러스터에 직접 배포하면(예: `kubectl apply`로 임의 이미지 지정) **아무것도 막지 못합니다.**

**Admission Controller가 최후 방어선**입니다. "골든 이미지에서 파생되지 않은 이미지는
클러스터에 뜰 수 없다"를 기계가 강제해야 합니다.

### 5-3. 빌드 이미지 거버넌스

발견 ③이 드러낸 문제입니다. 현재 구조:

```
goldenns/golden-base    ← 런타임 베이스 (통제됨)
golang:1.25             ← 빌드 이미지 (수동 관리)
```

개선안:
```
goldenns/golden-base    ← 런타임
goldenns/golden-go      ← 빌드용 (동일 파이프라인으로 관리)
```
앱 CI가 **두 `FROM` 모두** 검사하도록 확장합니다.

### 5-4. 스캐너의 본질적 한계

> **스캐너는 "알려진 것"만 찾습니다.**

| 상황 | 탐지 |
|------|------|
| 알려진 CVE + DB에 있음 | ✅ |
| 알려진 CVE + DB에 없음(EOL) | ❌ (발견 ①) |
| 알려지지 않은 취약점(zero-day) | ❌ |

**"스캔 통과 = 안전"이 아닙니다.** "스캔이 아는 범위에서 문제를 못 찾았다"일 뿐입니다.
따라서 스캔은 다층 방어의 한 층일 뿐이며, 화이트리스트·최소 이미지·최소 권한과 함께 사용해야 합니다.

## 부록: 검증 명령

### 승격 정확성 확인 (동일 digest인가)
```bash
kubectl get pod -n dev -o jsonpath='{.items[0].status.containerStatuses[0].imageID}'; echo
kubectl get pod -n qa  -o jsonpath='{.items[0].status.containerStatuses[0].imageID}'; echo
# 두 값이 동일해야 재빌드 없는 승격
```

### 환경별 설정 분리 확인
```bash
kubectl exec -n dev deployment/cicdapp -- printenv ENV_NAME   # development
kubectl exec -n qa  deployment/cicdapp -- printenv ENV_NAME   # qa
```

### 자동 배포 여부 검증 (교차 검증)
| 증거 | 확인 방법 | 강도 |
|------|-----------|------|
| Actions 로그의 배포 단계 | `deployment.apps/... created` + `successfully rolled out` | 결정적 |
| 워크플로에 배포 단계 존재 | `grep "Deploy to dev" .github/workflows/*.yml` | 결정적 |
| Deployment 이미지가 digest 형태 | `kubectl get deploy -o jsonpath='{..image}'` | 강함 |
| rollout REVISION | `kubectl rollout history` | 강함 |
| digest 일치 | Actions 로그 push digest = 파드 실행 digest | 강함 |

### 골든 이미지 확인
```bash
aws ecr describe-images --repository-name goldenns/golden-base \
  --region ap-northeast-2 --query 'imageDetails[].[imageTags[0],imageDigest]' --output table
# 스캔에 실패한 이미지는 애초에 push되지 않으므로, 이 목록에 있는 것은 전부 승인된 것
```

## 부록: 설계 즉답 체크리스트

- [ ] 머지하면 배포되나? → **아니오. 머지는 main 통합, 배포는 CD가 별도로**
- [ ] CD를 시작시키는 것은? → **머지 이벤트를 트리거로 파이프라인 자동 실행**
- [ ] 왜 QA를 DEV에 못 붙이나? → **코드 프리징. DEV는 머지마다 바뀌어 재현 불가**
- [ ] QA 배포 트리거는? → **사람이 태그를 찍는 것. '완료' 판단은 기계가 못 함**
- [ ] 승격 단위가 태그가 아닌 이유는? → **태그는 재지정 가능, digest는 내용을 보장**
- [ ] 이미지 1바이트 바뀌면 digest는? → **완전히 다른 값 (눈사태 효과)**
- [ ] 환경별 차이값은 어디서? → **ConfigMap. 이미지는 동일, 설정만 교체**
- [ ] 골든 이미지 관리 = 버전 관리? → **아니오. 승인·안전성·진위·투명성의 통제**
- [ ] 스캔 결과 0건 = 안전? → **아니오. EOL은 탐지 불가로 0건이 나옴**
- [ ] 게이트를 검증하는 방법은? → **실패 케이스. 통과는 아무것도 증명하지 않음**
- [ ] CI에 클러스터 자격증명을 주는 것이 위험한 이유는? → **러너 탈취 = 클러스터 탈취**
- [ ] 미완성 기능을 main에? → **feature flag로 꺼둠 (deploy ≠ release)**
- [ ] 대규모에서 trunk-based인 이유? → **merge hell 회피 + 잦은 통합**
